# Cloud Rental Feasibility — AWS / Alibaba Cloud

The owner buys no mining hardware; only **rented capacity** is considered, and
only on paper. This document records how the platform decides whether rented
AWS/Alibaba infrastructure is *permitted, measured, and economical* — and what
today's honest answer is. Nothing here creates resources or authorizes spend.

## Official sources (snapshot + hash + human review, mandatory)

Every policy decision requires a captured snapshot of the current version of
these sources — URL, capture date, SHA-256 of the captured text — reviewed by a
human. Registering a URL is not a legal opinion; this platform never interprets
law.

**AWS**
- Service Terms (incl. §1.25 obligations): <https://aws.amazon.com/service-terms/>
- Written-approval requirement for mining; unapproved mining treated as abuse:
  <https://aws.amazon.com/blogs/security/detecting-and-preventing-crypto-mining-in-your-aws-environment/>
- On-demand price source (Price List): <https://docs.aws.amazon.com/awsaccountbilling/latest/aboutv2/price-changes.html>
- Spot price source (the ONLY one): <https://docs.aws.amazon.com/AWSEC2/latest/APIReference/API_DescribeSpotPriceHistory.html>
- Accelerated instance specs: <https://docs.aws.amazon.com/ec2/latest/instancetypes/ac.html>

**Alibaba Cloud**
- Security lock — cryptocurrency mining listed as a security-violation example:
  <https://www.alibabacloud.com/help/en/ecs/developer-reference/api-behavior-when-an-instance-is-locked-for-security-reasons>
- Price source (DescribePrice): <https://www.alibabacloud.com/help/en/ecs/developer-reference/api-ecs-2014-05-26-describeprice>
- GPU instance families: <https://www.alibabacloud.com/help/en/ecs/user-guide/gpu-accelerated-compute-optimized-and-vgpu-accelerated-instance-families-1>

## Fail-closed posture (encoded in `cloud_rental/policy.py`, tested)

- **AWS `HASHING_WORKER`** → `BLOCKED_PENDING_WRITTEN_APPROVAL` unless a
  verifiable, human-reviewed, unexpired AWS Trust & Safety written-approval
  artifact exists. Free Tier / promotional credits: always blocked.
- **Alibaba `HASHING_WORKER`** → `BLOCKED_PROVIDER_POLICY` by default. Only an
  explicit, current, human-reviewed **written contractual exception** could
  change it — a ticket or an assumption is not enough.
- **`CONTROL_PLANE` / research / paper** on either provider: ordinary compute;
  evaluable offline against a budget ceiling — still with no resource creation.
- Missing, ambiguous, expired, or unattributable policy evidence →
  `BLOCKED_POLICY_UNKNOWN`. A BLOCKED status keeps its legal/operational reason
  and is never silently converted into an economic NO-GO.

## Benchmark rule (encoded in `cloud_rental/benchmarks.py`, tested)

A quote has no hashrate. A `hashrate_hs` typed into YAML is rejected. Evidence
means a real measured benchmark of the **exact SKU** (provider, accelerator
model and count, algorithm, ≥10 min after warmup, shares accepted/rejected,
capture date, artifact hash). No extrapolation between GPU models, between
SKUs, or GPU→ASIC. SHA-256 on general CPU/GPU without such a benchmark is
`BLOCKED_INCOMPATIBLE_HARDWARE`; "EC2 = S21" equivalences do not exist.

## Economics rule (encoded in `cloud_rental/economics.py`, tested)

Rental flows are cancelable hourly flows: revenue per **useful** compute hour
(measured hashrate × point-in-time hashprice, minus pool fee, rejects,
utilization, interruption/checkpoint loss) versus **all-in** rented cost per
hour (price × FX × (1+VAT) + disks + IP + logging + egress). Margins are
reported at 1×/2×/3× costs with a break-even hourly price, bounded by an
explicit budget ceiling. **No multi-year NPV is applied to a spot instance** —
the calculator refuses horizons beyond ~one quarter. Electricity is never
double-charged: it is inside the rental price.

## Current matrix (offline example configs; quotes are FIXTURES, not prices)

Generated by `quant-trade cloud-rental compare` over
`configs/cloud_rental/*.example.yaml`:

| Provider | Purpose | Status | Reason |
| --- | --- | --- | --- |
| AWS | control_plane | **PAPER_CONTROL_PLANE_CANDIDATE** | ordinary compute; ~$0.03/h fixture within a $50/month ceiling |
| AWS | hashing_worker | **BLOCKED_PENDING_WRITTEN_APPROVAL** | no written Trust & Safety approval artifact on file (Service Terms §1.25) |
| Alibaba | control_plane | **PAPER_CONTROL_PLANE_CANDIDATE** | ordinary compute; ~$0.04/h fixture within a $50/month ceiling |
| Alibaba | hashing_worker | **BLOCKED_PROVIDER_POLICY** | mining is a security-violation lock example; no written contractual exception |

**Honest reading:** both providers are fit to host the *software* (collectors,
research, dashboards, paper runtime) subject to budget; **neither provider is
currently eligible for hashing**, before economics are even reached. That is a
valid, expected result. Additionally, even if policy were satisfied, no
measured benchmark of any cloud SKU exists in this repo, so the hashing rows
would then be `BLOCKED_MISSING_BENCHMARK` — and a measured ~20 GH/s GPU
benchmark against a $10/h SKU prices out at fractions of a cent of revenue per
hour (`ECONOMIC_NO_GO`, tested).

## What would change these answers

1. AWS: a written Trust & Safety approval naming the account and workload
   (snapshot + hash + expiry + human review) → unblocks *evaluation only*.
2. Alibaba: a written contractual exception (same evidentiary bar).
3. A real benchmark of the exact SKU, captured as a hashed artifact.
4. Fresh quotes from the real price sources (Price List / Spot history /
   DescribePrice — never mixed).
5. Positive margins at 1× and 2× costs within the budget ceiling.

Each step unblocks only the next evaluation stage. None of them authorizes
resource creation or spend — `EXTERNAL_SPEND_AUTHORIZED: FALSE` stands.
